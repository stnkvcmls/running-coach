Redesign mockups — target end state

Static, high-fidelity mockups of the screens specified in docs/UI_UX_REDESIGN_PLAN.md, shown as before/after pairs: the current implementation (grey badge, issues tagged in red) beside the redesign target (phase badge). Both sides use the same sample data so differences are design, not data. Frames use the app's real design tokens (globals.css + Phase 0 additions). Toggle component labels to cross-reference each element against the plan's phase tasks after implementation.

Today

From a flat 11-section card stack (readiness, briefing, and the day's session never meet; check-in stays full-size after submit) to a single hero that answers "what am I doing and am I ready?" above the fold.

Plan

From uniform day cards with no completion state to a week checklist: ✓ done (linked to the run), ✗ missed, ▶ today, week tabs that carry volume and status.

Activities

From one-line text rows grouped by month to scannable cards: sport icon, PB badge, zone micro-bar, weekly group totals. Both frames in light theme to spec that palette.

Activity detail

From a 2,000-px scroll of stat grids with a table for splits and the AI verdict at the very bottom, to a page with wayfinding: sticky header, verdict up top, splits you can read at a glance, deep stats collapsed but intact.

Progress (Trends)

From six cramped tabs on a transparent bar (the undefined --surface token) with per-chart palettes, to chip tabs, a shared range selector, one chart theme, and records that celebrate.

Desktop ≥ 1024 px

From a centred 640-px phone column with dead margins to a nav rail and a two-column Today grid — CSS-only reflow of the same components.

How to use this file

  • For review: each surface shows the current implementation (grey "Current" badge; known issues tagged in red with audit-section references) beside the redesign target (phase badge). Both sides use the same sample data. The "after" frames are the visual target for each phase — layout, hierarchy, and component states. Copy is realistic sample data, not final copy.
  • For implementation cross-reference (Sonnet): after finishing a phase, open this file with component labels on and compare the built UI against the matching frame. Labels name the component file and plan task (e.g. TodayHero.tsx · task 2.1). Match structure and hierarchy, not pixel positions.
  • Tokens: frame styling uses the values from frontend/src/styles/globals.css plus Phase 0 additions (updated --text-muted #98a0b3 dark / #5b6472 light, --surface, tabular numerals). Zone colours in micro-bars are illustrative — implementation uses MetricZone.zone_color from the API.
  • Deviations are allowed where the plan text and mockup disagree: the plan document (docs/UI_UX_REDESIGN_PLAN.md) is authoritative; this file illustrates it.
